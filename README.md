# 🍽️ Telegram Menu Planning Bot

Telegram-бот, который помогает собирать рацион питания под пользователя и превращать его в список покупок. Рецепты живут в SQLite (сейчас 143 рецепта в 4 категориях), состав и КБЖУ известны, поэтому бот умеет считать покупки по выбранным блюдам, а в будущем — подбирать рацион под целевую калорийность и имеющиеся дома продукты.

Полный продуктовый roadmap с приоритетами, стадиями готовности и известными проблемами — в `/.claude/plans/product-roadmap.md`. Этот README — краткая выжимка для общего понимания.

## Возможности

Обозначения: ✅ работает, 🟡 частично, 🔴 запланировано.

### Основное

- ✅ **Рандом блюда по категории** — выбор завтрака / основного / салата / десерта с карточкой (название, КБЖУ, ингредиенты, описание) и возможностью «ещё вариант» / «выбрать».
- 🟡 **Добавление блюда в рацион на сегодня** — работает, но слот выбранного блюда пока определяется порядком добавления, а не категорией (баг в разработке, см. roadmap).
- 🟡 **Просмотр текущего рациона** — сгруппирован по датам, показывает блюда по дням и категориям. Ограничение по текущей неделе пока не применяется.
- 🟡 **Список покупок** — агрегация ингредиентов из рациона с группировкой по типам продуктов. Окно выборки (какой период считать) сейчас настроено некорректно — фикс в roadmap.
- 🔴 **Генерация случайного рациона на неделю** — спроектирована, заглушка в коде. Должна уметь учитывать исключённые ингредиенты и целевую калорийность.
- 🔴 **Правка конкретного приёма пищи в недельном рационе** — через кнопку рядом с блюдом: заменить / удалить / оставить.
- 🔴 **Список покупок на выбор — день или неделя** — UI-переключатель и разные окна агрегации.
- 🔴 **Очистка рациона с подтверждением** — сейчас «Очистить день» / «Очистить список» удаляют сразу, нужен двухшаговый confirm.

### Персонализация

- 🔴 **Анкета пользователя** — вес, рост, возраст, пол, уровень активности, целевая калорийность. Таблица `user_preferences` уже есть в БД (пока используется только заготовка под `exclude_ingredients` и `target_calories`).
- 🔴 **Список исключений** — ингредиенты, которые бот никогда не должен предлагать (аллергии / непереносимости). Поле в БД есть, логика учёта в подборе не написана.

### Дальние планы

- 🔴 **Расчёт КБЖУ по формуле Миффлина-Сан Жеора** — выводит целевые калории под цель (похудение / поддержание / набор) и записывает в `user_preferences.target_calories`.
- 🔴 **Обратный поиск «что приготовить из моих продуктов»** — пользователь вводит список имеющихся ингредиентов, бот возвращает подходящие рецепты. Требует предварительной нормализации справочника ингредиентов.
- 🔴 **Уровни пользователей и подписки** — часть функций (недельная генерация, обратный поиск, расчёт КБЖУ, экспорт) планируется как платная. Тарификационная модель ещё не определена.

### Backlog (идеи на рассмотрение)

См. полный список в roadmap. Коротко: теги рецептов (vegan / gluten free / quick), избранные, пользовательские рецепты, дневник реального питания, интерактивный список покупок с чекбоксами, экспорт в PDF/CSV, утренние напоминания, порционирование для семьи, админские команды для управления рецептами, inline-режим, мультиязычность.

## Известные проблемы

- **Задвоение ингредиентов в списке покупок.** В справочнике встречаются «Лук» и «Лук репчатый», «Помидор» и «Помидоры» и т.п. — они считаются разными записями, поэтому один и тот же продукт может появиться в списке несколько раз. Решения обсуждаются в roadmap (нормализация в `migrate.js`, словарь алиасов, таблица `ingredient_aliases`).
- **Все ингредиенты сейчас с `type='general'`.** Из-за этого UI-группировка по типам (овощи / молочка / мясо / …) не работает: всё сваливается в «Прочее». Типизация будет добавлена одновременно с нормализацией.
- **Фильтр по неделе в SQL некорректный.** `getIngredientsFromPlan` берёт прошлую календарную неделю вместо текущей; `getWeeklyPlan` берёт все записи пользователя. Ограничения по периоду будут приведены в порядок в ближайшей итерации.

## Технологии

- [Telegraf](https://github.com/telegraf/telegraf) — Telegram Bot Framework
- [sqlite3](https://github.com/mapbox/node-sqlite3) — SQLite bindings
- [winston](https://github.com/winstonjs/winston) — логирование
- [dotenv](https://github.com/motdotla/dotenv) — загрузка переменных окружения

## Запуск

Требования: Node.js ≥ 14, npm, действительный токен Telegram-бота от BotFather.

1. Установить зависимости:
   ```bash
   npm install
   ```
2. Создать `.env` по образцу `.env.example`:
   ```env
   BOT_TOKEN=ваш_токен_от_botfather
   DATABASE_PATH=./database/bot.db
   LOG_LEVEL=info
   ```
3. Залить рецепты в SQLite (создаёт все таблицы идемпотентно и вставляет рецепты из `database/recipes.json`):
   ```bash
   node src/database/migrate.js
   ```
4. Запустить бота:
   ```bash
   npm start        # продакшн
   npm run dev      # с автоперезагрузкой через nodemon
   ```

## Структура проекта

Правила модульной организации зафиксированы в `projectRules.mdc` и продублированы в `CLAUDE.md`. Коротко:

- `src/bot/` — константы и локализация (`texts.js`). Все пользовательские строки живут только здесь.
- `src/interface/` — UI-слой: клавиатуры и форматирование сообщений (один файл на экран).
- `src/features/<feature>/logic.js` — бизнес-логика и работа с БД. Сейчас есть три фичи: `meal-suggestions`, `weekly-planner`, `shopping-list`.
- `src/database/` — подключение к SQLite (`db.js`) и миграция из JSON (`migrate.js`).
- `src/utils/` — общие утилиты (пока только `logger.js`).
- `database/` — исходный файл рецептов (`recipes.json`) и сам SQLite-файл (`bot.db`).
- `logs/` — файлы логов winston (`error.log`, `combined.log`).

Точка входа — `src/index.js`: там регистрируются все Telegraf-хэндлеры (`bot.hears` / `bot.action`).

## Схема базы данных

Вся схема создаётся в `src/database/db.js:initDb` идемпотентно через `CREATE TABLE IF NOT EXISTS`. Перечень таблиц:

1. `users` — `id`, `telegram_id`, `username`, `created_at`. Внутренний `id` используется во всех FK.
2. `meal_categories` — справочник категорий (`breakfast`, `main`, `salads`, `desserts`).
3. `recipes` — название, категория, описание, КБЖУ на порцию.
4. `ingredients` — уникальные ингредиенты и их тип (`vegetable`, `dairy`, …; сейчас все `general`, см. «известные проблемы»).
5. `recipe_ingredients` — связка рецептов и ингредиентов с количеством и единицей измерения.
6. `daily_menu` — рацион на конкретный день пользователя, уникальность `(user_id, date)`.
7. `daily_menu_items` — блюда в рамках дня, поле `slot` описывает позицию: `1=завтрак`, `2=основное №1`, `3=основное №2`, `4=салат`, `5=десерт`.
8. `weekly_menu` — недельный рацион, уникальность `(user_id, week_start)`.
9. `weekly_menu_days` — связка `weekly_menu` с конкретными `daily_menu`.
10. `user_preferences` — `exclude_ingredients` (JSON) и `target_calories`. В будущем расширится полями анкеты (вес, рост, возраст, пол, активность).
11. `shopping_lists` + `shopping_list_items` — сохранённые списки покупок за период.

Подробные комментарии к каждой таблице и их обоснование — в `.claude/plans/product-roadmap.md` и в самом файле миграции.
